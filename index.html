Unity 에디터 확장·개발 편의 기능

개발 과정의 반복 작업과 비효율을 개선하고, 프로젝트 특성에 맞는 유틸리티와 Unity 에디터 확장 기능을 구현하여
팀의 작업 효율과 생산성을 높입니다

Attribute 기반의 자동 이벤트 바인딩 시스템

UIToolkit 환경에서, element class를 통해 이벤트를 자동으로 바인딩하는 기능을 구현하였습니다. 디자이너는 UI를 새로 만들거나 수정할 때, element class를 추가하면 즉시 기능을 적용할 수 있으며, 프로그래머는 Attribute를 선언하는 것으로 추가 바인딩 작업 없이 이벤트를 바인딩할 수 있습니다.

[ElementCommand("cmd--button--brush-tool")]
private static void SelectBrushTool(VisualElement target) => History.Log("브러쉬 도구를 선택하였음");

UQuery를 통한 리플렉션 호출을 초기화 시점에 단 한 번으로 최적화하여 성능 영향을 최소화했습니다. static 람다를 사용해 외부 변수 캡처로 인한 부가적인 가비지 컬렉터(GC) 할당을 방지했습니다.

Naming Tool

지정된 네이밍 규칙에 맞추어 하이어라키 내 오브젝트 및 프로젝트 에셋의 파일명을 자동으로 변경해주는 에디터 확장 기능을 구현하였습니다. 컴포넌트 구성의 우선순위를 분석하여 이름을 생성합니다.

컴포넌트 기반 네이밍

게임 오브젝트에 부착된 컴포넌트의 특성을 다각도로 분석하여 이름을 결정합니다. 단순히 컴포넌트 형식명만 부여하는 것이 아니라, 컴포넌트 내부의 멤버 변수(참조 중인 메쉬 데이터, 렌더러 재질 등) 정보를 정교하게 읽어와 네이밍 규칙에 맞게 반영합니다. (예: MeshRenderer 감지 시, MeshFilter의 메쉬명과 Material명을 결합하여 고유 네이밍을 자동 생성)

넘버링 시스템

유니티 엔진에서 에셋 중복 생성 시 무분별하게 부여하는 디폴트 넘버링 패턴을 감지하여 프로젝트 표준 포맷으로 즉시 보정합니다. 또한 계층 구조 내 자식 객체들의 순서를 분석해 정렬된 인덱스 넘버링을 적용함으로써 복잡한 하이어라키 뷰의 시각적 정돈과 휴먼 에러 차단을 도모했습니다.

에셋 타입 기반 유효성 검증

선택된 에셋의 바이너리 타입(Texture, AudioClip, Material 등)을 판별하여 사전에 정의된 고유 접두사(Prefix) 규칙을 자동 매핑합니다. 정규표현식을 활용하여 공백이나 허용되지 않는 특수문자 등의 예외 규칙을 걸러내고, 유효하지 않은 문자는 자동 제거 또는 사전 합의된 규칙 문자로 안전하게 치환합니다.

확장성을 고려한 아키텍처 설계

네이밍 로직 전체를 Abstract Class로 정의하고, Light, Texture 등 개별 에셋/컴포넌트 타입이 이를 구현하여 상속받도록 결합도를 낮추었습니다. OCP(개방-폐쇄 원칙)를 준수하여 새로운 에셋 규격이나 컴포넌트 타입이 추가되더라도 기존 코드를 수정할 필요 없이, 해당 네이밍 구현체를 추가 정의하는 것만으로 손쉽게 기능 확장이 가능하도록 설계했습니다.

CSV Utility

별도의 파싱 코드 작성 없이 CSV 데이터를 C# 컬렉션 및 오브젝트로 자동 변환하는 컨버터 유틸리티를 구현했습니다. 헤더 이름 매핑과 열 인덱스 매핑을 모두 지원합니다.

public class Weapon
{
    [CSVIndex(0)] public string ItemCode;
    [CSVHeader("wpn_author")] public string AuthorName;
    public List<string> EnchantList;
    public EquipSlotType SlotType;
}

컨버터 클래스를 상속하여 복잡한 자료형(2중 딕셔너리 등)도 CSV 포맷 안에서 직접 구조화하여 파싱할 수 있게 지원합니다.

레벨 에디팅 및 베이킹 툴

씬 뷰에서 Handles와 Gizmos를 활용하여 AI 이동 경로 노드를 직접 생성하고, 지형 데이터를 가공할 수 있는 레벨 에디터를 개발했습니다. NavMesh 에셋을 직접 구동하지 않고 ScriptableObject로 구역 좌표 데이터를 압축 베이킹하는 툴을 제공했습니다.

Addressables 시스템 확장

프로젝트의 Addressables 적용 방식에 맞춘 커스텀 래퍼 클래스와 커스텀 인스펙터를 구현했습니다. 이를 바탕으로 에셋 참조 규칙과 라이프사이클 관리 구조를 정비하고, Global/Stage 단위 참조 카운팅으로 에셋의 로드·언로드 흐름을 관리했습니다. 예제가 포함된 가이드라인을 공유해 팀원이 일관된 방식으로 에셋을 적용할 수 있도록 지원했습니다.

게임 플레이에 필요한 콘텐츠와 다양한 기능 요구사항을 구현합니다

  • StateMachineBehaviour 기반의 상태별 이벤트 제어로 애니메이션과 보스 패턴 로직을 1:1로 매핑

트러블슈팅

지형 생성 시간 단축 및 메모리 최적화

GC Allocation 제거 및 청크 기반 연산 최적화를 통해 맵 생성 시간을 96% 단축했습니다.

문제 상황

기획상 요구되는 최대 크기 환경에서 거대한 레벨을 생성할 경우, 300초 이상의 긴 시간이 소모되는 현상이 발생하였습니다.

원인 분석

Profiler를 사용하여 분석한 결과, 심각한 GC 오버헤드와 비효율적인 연산이 핵심 원인으로 파악되었습니다.

  1. GC 오버헤드:
    • class 기반 좌표 클래스의 연산자 오버로딩으로 인해 매 연산마다 발생하는 힙 할당
    • ToList, ToDictionary 등의 잦은 LINQ 호출로 인한 리사이징과 박싱
    • 잦은 Coroutine 호출
  2. 비효율적 알고리즘:
    • 오버랩 체크 시 타일 개수만큼 객체를 생성하거나 전체를 순회하는 O(N^2) 로직
    • 동일한 결과를 반환하는 조건 연산의 중첩 사용

해결 과정

1. GC 오버헤드 제거

좌표 클래스를 구조체로 변경하는 동시에 지형 정보를 담고 있던 string 등 참조 타입의 변수를 제거하여 GC.Alloc을 0B로 유지할 수 있게 하였습니다.
또한 LINQ 사용으로 인해 발생하는 문제점을 해결하기 위해 아래와 같이 수정하였습니다.

  • 컬렉션을 미리 생성함과 동시에 Capacity를 계산하여, 함수 내에서의 생성과 Resizing을 최소화하였습니다. (Memory Profiler를 통해 예약된 메모리와 컬렉션의 최종 크기가 동일함을 검증했습니다.)
  • 쿼리 함수를 for/while를 사용한 순회로 변경하여 새로운 컬렉션을 생성하지 않도록 하였습니다.
  • List 순회 대신 HashSet과 Dictionary을 사용하여 평균 데이터 접근 속도를 개선하였습니다.

마지막으로 코루틴은 컴파일 단계에서 클래스로 선언되어 작동하기 때문에 GC.Alloc이 일어나게 되므로, 이를 방지하기 위해 구조체 기반의 UniTask를 도입하였습니다.

2. 청크 시스템 적용

청크 시스템의 Spatial Hash Grid를 적용하여 좌표 단위로 실행하는 동작을 영역 단위로 실행하도록 수정하였습니다.
오버랩 체크, 영역 회전, 확장 등 관련 함수의 호출 횟수와 평균 데이터 접근 속도를 개선하였습니다.

3. CPU Friendly

개선된 구조에서도 일부 함수는 최악의 경우 수십만번 이상 호출되어 실행 시간에 영향을 주는 것을 확인하였습니다. 더 이상 호출 횟수를 줄이는 것은 어려웠기 때문에, 함수 자체의 동작 시간을 줄이기 위해 아래와 같은 방법을 통해 CPU의 처리 시간 감소를 도모하였습니다.

  • 좌표 구조체의 크기를 조절하여 메모리 접근 횟수를 줄였습니다.
  • 불필요한 중간 객체를 제거하고 함수 파라미터에 수식을 직접 넣어 전달하였습니다.
  • 연산자 오버로딩을 제거하고 멤버의 값을 수정하는 연산 함수로 변경하였습니다.

결과

최악의 케이스(모든 청크 검사 실패, 기획상 최대 지형 크기 적용 등)를 가정한 테스트 환경에서, 평균 생성 시간이 380초 → 14초로 단축되었습니다.
또한 Profiler로 측정 결과, 핵심 생성 로직에서의 ZeroAllocation을 달성하고, GC Spike 현상을 개선할 수 있었습니다.

프로젝트 내 NavMesh 작업 프로세스를 유지한 채 WheelCollider를 사용하기

NavMesh를 사용하는 월드에서 작동하는 WheelCollider를 이용한 주행 기능을 구현하였습니다.

문제 상황

개발 도중 플레이어의 이동 방식이 NavMeshAgent에서, 물리가 적용된 WheelCollider로 변경되었습니다.
하지만 기존 스테이지는 NavMesh 규격으로 이미 제작되어 수정이 불가능한 상황이었으며, 단순히 NavMeshAgent를 사용할 경우 물리적 상호작용(충돌, 서스펜션 등)을 구현하는 데 한계가 있었습니다.
따라서 기존 NavMesh 데이터를 활용하면서도, 물리 엔진 기반의 주행이 가능한 시스템을 새롭게 구축해야 했습니다.

해결 과정

1. NavMesh 데이터 추출 및 AI Navigation 동작 분석

Unity AI Navigation Package의 Baking 함수를 추적하여, 베이킹 시 생성되는 NavMesh 폴리곤의 좌표 데이터를 추출하는 기능을 구현했습니다.
추출된 데이터를 별도의 ScriptableObject로 저장하여, 런타임에 경로 데이터를 참조할 수 있도록 구조화했습니다.

2. 주행 경로 수정

NavMesh가 지원하는 경로 따라가기 함수 사용 시 폴리곤의 꼭짓점 부분을 이용하여 이동하기 때문에 시각적으로 부자연스러운 문제가 발생하였습니다.
이를 해결하기 위해 삼각형의 무게중심을 선형 보간한 경로를 사용하였으나, 삼각형 형태에 따라 불필요한 움직임이 발생했습니다.
최종적으로는 인접한 폴리곤이 맞닿는 모서리의 중점을 연결하여 경로를 수정하는 로직을 적용하여, 차량이 도로의 중앙을 따라 부드럽게 이동하도록 구현했습니다.
또한 Box, Capsule 등 장애물을 감지하면 Raycast를 통해 우회 경로를 생성하도록 구현했습니다.

결과

NavMeshAgent를 직접 제어하지 않고, AI Navigation의 AutoBraking, OffMeshLink, AreaMask 동작을 분석해 WheelCollider 차량 주행 로직으로 재구현했습니다. 기존 NavMesh 데이터를 경로 생성에 활용하여, 물리 차량의 자율 주행을 구현했습니다.
