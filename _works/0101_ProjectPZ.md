## 절차적 레벨 생성
미리 만들어진 작은 지형 데이터의 집합으로 영역을 만들고, 영역의 집합으로 하나의 레벨을 만들어내는 로직을 구현하였습니다.  
영역과 영역은 인접 가능한 지점이 정해져있고, 해당 지점을 통해서만 이동이 가능한 구조로 생성되어야 했음  
이는 절차적 텍스처 합성에 사용되는 Wang Tile을 응용하여 실현 가능하다고 판단, 생성 로직을 구상 후 문서로 작성하여 디자이너와 나의 뇌를 동기화.

# Stuck & Strike (Troubleshooting)

[Zero allocation과 propagation를 통한 성능 개선](#){: .btn-sub sub="절차적 레벨 생성 과정에서의 문제 발생과 해결"}
### 문제 발생
아니 어느 순간부터 맵 한번 만들어지는거 보려면 무슨 8분 10분을 기다려야하더라고요?  
이게 뭐 특정 컴퓨터에서만 오래걸리는게 아니라 뭐 뭘 써도 다 똑같이 오래걸림  
벌써부터 이렇게 오래걸리면 진짜 미래가 걱정되는 수준 아니겠음?...  
일단 재미검증용으로 작성된 프로토타입 코드를 리팩토링하지 않고 그대로 사용했기에 예상하고 있던 문제긴 함
### 원인 파악
Profiler 분석 결과 심각한 GC 오버헤드(GC.Alloc, GC.Collect)와 자주 호출되는 함수의 긴 동작 시간으로 인한 문제임을 파악하였음.  
이후 ProfilerMarker를 통해 어떤 부분에서 GC 오버헤드가 발생하는지, 또 함수의 어떤 부분이 동작 시간을 증가시키는지를 알아냈음.  

1. 불필요한 객체 생성으로 인한 GC 오버헤드
    - 좌표 클래스의 연산자 오버로딩으로 인해 매 연산마다 새로운 객체 생성
    - 다음 지형 생성 위치를 선정하기 위한 오버랩 체크 시 타일의 개수만큼 새로운 객체 생성
    - 결과물 가공 후 버려지는 데이터들의 처리
2. LINQ 사용으로 인한 GC 오버헤드
    - ToList, ToDictionary등의 함수로 인한 새로운 객체 생성 및 반복적인 Resizing
    - 콜렉션의 Enumerator가 LINQ용 인터페이스로 캐스팅
3. Coroutine 사용으로 인한 GC 오버헤드
4. 지형 생성 위치 선정 함수의 긴 동작 시간

### 문제 해결

#### CPU Friendly
- 좌표 클래스를 좌표 구조체로 변경 후, 크기를 8byte로 줄여 CPU 처리 속도 개선
- 불필요한 중간 객체를 제거하고 함수 파라미터에 직접 계산식을 넣어 CPU 처리 속도 개선

#### 구조 재설계
- 지형 생성 위치 선정 로직 수정
    - 런타임 최적화에 사용하던 청크 구조를 도입하여 모든 과정의 평균 연산 속도 개선
    - List 순회 대신 HashSet과 Dictionary을 사용하여 데이터 접근 속도 개선
- 결과물 가공 프로세스 제거, 되도록 필요한 데이터만 사용하도록 개선

#### GC 오버헤드 제거
우선 LINQ 쪽을 해결 
- 사용할 콜렉션은 미리 생성해두고, 생성 시 크기를 미리 예측하여 설정함
- for/while을 사용한 순회로 새로운 콜렉션을 생성하지 않도록 함
그리고 불필요한 객체 생성 해결
- 연산자 오버로딩을 제거하고 연산용 함수로 변경
- 좌표 클래스를 좌표 구조체로 변경하여 GC 해방
- 구조 재설계 과정에서 일부 지형 클래스의 생성을 단순 계산식으로 수정
그리고 코루틴 쓰는 부분들 UniTask로 바꿨음

### 결과
캐싱을 위한 객체 생성 시 발생하는 필수적인 할당을 제외하면, 문제가 되는 부분에서 Zero Allocation을 달성하였음.  
맵 생성 속도가 얼마나 개선되었는지를 측정하기 위해  
최악의 경우를 가능한 구현(모든 청크 검사 실패, 일부 반복문 강제 순회, 기획상 최대 지형 크기 적용 등)하여 측정한 결과,  
기존 코드는 평균 380초의 동작시간, 개선한 코드는 평균 24초의 동작시간을 기록하였습니다.
